# 我如何用纯 AI 构建并部署这个个人网站

这篇文章是对本站构建过程的一次复盘。

这个项目有一个很明确的前提：我希望尽量不手写业务代码，而是把 AI 当成主要工程执行者，从需求拆解、代码实现、调试测试，到最终部署到自己的 NAS，都由 AI 持续推进。我负责提出目标、判断体验、给出反馈，以及在关键节点确认方向。

它最后变成了一个 Next.js 个人站点：既有博客，也有文件管理、日记、待办、AI 对话、订阅摘要、图片生成、技能系统、命理工具和后台管理。更重要的是，它不是一个只停留在本地的 demo，而是已经跑在我自己的 NAS 上，能持续更新、测试和迭代。

## 一开始我想要的不是一个“页面”，而是一个自己的工作台

最初的目标并不复杂：我需要一个属于自己的站点，可以写博客，也可以承载一些日常工具。

但真正开始构建后，需求很快变成了一个更完整的个人工作台：

- 博客文章可以用 Markdown 管理。
- 后台可以登录，管理文章、文件、日记、待办和订阅。
- AI 能力可以接入 Claude 和 GPT，并且支持流式输出。
- AI 对话要有历史记录、删除功能、Markdown 渲染和更好的阅读体验。
- 订阅源可以跟踪 AI 相关博客、GitHub、X、RSS，并生成摘要。
- 项目要能部署到 NAS，而不是依赖本地开发机。

如果用传统方式做，这会是一组不断膨胀的小需求。纯靠 AI 构建时，我做的第一件事不是让它“写个网站”，而是不断把需求拆成可验证的小块：先能跑，再能用，再变好用。

## AI 在这个项目里承担的是工程同伴，而不是代码补全

这个项目让我感受最深的一点是：AI 最适合承担“持续推进”的角色，而不是只回答某个孤立问题。

每次我提出一个体验问题，AI 都会先读现有代码，再判断应该改 API、数据库、组件还是部署脚本。例如：

- AI 对话无法真实请求返回值，就检查 provider 配置、流式接口和上游 API 形态。
- GPT-5.5 的 Right Code 接口走 Responses API，就重写 OpenAI-compatible 的请求和 SSE 解析。
- Markdown 代码块黑底黑字，就定位到 `prose-code` 样式覆盖了深色代码块。
- 全屏聊天层级不对，就把聊天容器 portal 到 `document.body`，并提高 z-index。
- 订阅每次刷新都调用 AI，就拆成“每日抓取原始内容”和“按需整合摘要”两条链路。

这不是一次性生成代码，而是一种循环：

1. 我描述问题或新的使用感受。
2. AI 读代码和测试，找到真实入口。
3. AI 先补测试，证明旧行为有问题。
4. AI 修改实现。
5. AI 跑测试、构建、e2e。
6. AI 部署到 NAS，再做线上 smoke test。

这个循环让项目越做越像一个真实产品，而不是一堆功能拼图。

## 技术骨架选择得很克制

项目技术栈并不花哨：

- Next.js 14 App Router 负责页面和 API。
- SQLite 存储待办、日记、AI 聊天历史、订阅源和摘要。
- Tailwind CSS 负责界面样式。
- Vitest 覆盖 API 和核心逻辑。
- Playwright 覆盖关键前端流程。
- Docker Compose 部署到 NAS。

这个选择有几个好处。

第一，SQLite 对个人站点足够简单，不需要额外维护数据库服务。第二，Next.js 的 API Route 可以把前端、后台、AI 调用和订阅抓取都放在同一个应用里。第三，Docker 化之后，NAS 只需要构建镜像并运行 compose，不需要在 NAS 上手工配置 Node 环境。

AI 构建项目时，技术越克制越好。因为复杂栈会放大 AI 的误判空间，而简单栈更容易通过测试和日志把问题压回到现实里。

## AI 能力是逐步接进去的

AI 对话不是一开始就顺利的。

项目先有 Claude 配置，后来接入 Right Code 的 Claude 代理，再接入 Right Code GPT-5.5。这里最关键的变化是区分不同上游 API 的协议：

- Claude 使用 Messages API。
- GPT-5.5 使用 Responses API。
- 两者都需要支持流式输出。
- 前端只关心统一的 SSE 文本增量。

后来我又把 AI provider 配置临时收紧为 `.env.local` 只读模式。后台页面只负责展示和测试 Claude、GPT 两个环境 provider，不再开放新增、编辑、删除。这样做牺牲了一点灵活性，但换来了部署稳定性：生产环境到底在用哪个 key、哪个模型，一眼就能从环境变量和只读页面确认。

同时，项目里的 skills 也做成 provider-neutral。也就是说，同一份 skill prompt 不绑定 Claude 或 GPT，而是由后端根据当前 provider 选择正确的请求格式。这让 NAS 上的大模型能力可以更通用。

## 订阅功能从“每次问 AI”改成了“先抓取，再整合”

订阅功能是一次很典型的产品化迭代。

早期版本里，点击刷新订阅会抓取远端内容，然后立刻调用 AI 生成摘要。这能跑，但并不理想：每次刷新都消耗 AI，而且抓取和总结绑定在一起，失败时也很难判断到底是网络问题、RSS 问题，还是模型问题。

现在它被拆成两步：

- `crawl`：每天定时抓取 RSS、博客、GitHub、X、Reddit 等来源，把原始内容写入 `subscription_items`。
- `integrate`：当我需要看摘要时，再点击整合，由 AI 读取已经抓取好的内容生成 `subscription_briefs`。

NAS 上还跑了一个 `subscription-cron` 容器，每天自动调用抓取接口。这样，AI 不再负责“找资料”，而是负责“理解已经抓到的资料”。这个边界非常重要，因为它让系统更可控，也更省。

## 部署到 NAS 是这个项目真正落地的分界线

本地能跑只是第一步。真正让我觉得这个项目成立，是它被部署到了自己的 NAS 上。

部署链路也由 AI 一步步补齐：

- 写 Dockerfile，构建 Next.js standalone 应用。
- 写 `docker-compose.nas.yml`，同时运行 app、Claude worker、Cloudflare tunnel 和订阅 cron。
- 写 `deploy-to-nas.sh`，自动打包源码、上传 `.env.local`、远程构建镜像、重启 compose。
- 部署后自动清理远端 staging 目录，关闭 SSH/SFTP 会话。
- 每次大改动后都跑测试、构建、e2e，再部署和线上 smoke。

这个过程的价值不只是“上线了”，而是把发布变成了可重复动作。以后我再提出一个需求，AI 不只是能改代码，还能把它带到 NAS 上验证。

## 纯 AI 构建并不意味着人可以完全缺席

这个项目虽然是纯 AI 构建，但它并不是“人什么都不管”。

我的角色更像产品负责人和验收者：

- 我描述真实使用感受，而不是抽象功能清单。
- 我指出哪里“不对劲”，比如全屏在底层、Markdown 看不清、流式没有返回。
- 我决定一些取舍，比如暂时关闭 AI 配置写入，只保留环境配置。
- 我要求部署后再次测试，而不是本地通过就算完。

AI 的优势是执行密度很高，能连续读代码、改代码、补测试、部署、复测。但方向感仍然来自真实使用者。一个 AI 项目要做成，不是靠一次完美提示词，而是靠持续反馈。

## 我从这个项目里得到的经验

这次构建让我确认了几件事。

第一，AI 已经可以承担完整的小型全栈项目交付，但前提是项目有清晰边界、可运行测试和可重复部署流程。

第二，需求最好从真实体验出发。比如“Markdown 看不清”“页面不要自动滚动”“订阅不要每次都调用 AI”，这些问题比“优化一下 UI”更容易变成正确代码。

第三，AI 生成的系统必须被测试约束。没有测试，AI 很容易看起来完成了；有测试，它才会真的把行为对齐。

第四，把项目部署到自己的 NAS，是个人软件很好的落点。它不像云服务那样复杂，也不会停留在本地玩具状态。它是一个可以长期运行、不断被自己使用和改造的地方。

## 最后

这个网站现在还会继续进化。

它已经不只是一个博客，而是一个由 AI 帮我持续构建的个人系统。它能写作、记录、订阅、对话、管理文件，也能作为我继续实验 AI 工作流的根据地。

对我来说，这个项目最有意思的地方不是“AI 写了多少代码”，而是我第一次真切感到：一个人可以带着 AI，把一个想法从一句话推进到一个运行在自己 NAS 上的系统。

这件事本身，就很值得记录下来。
