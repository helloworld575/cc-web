# TDD 与测试门禁

本仓库对 Codex / agent 的默认要求不是“先写功能，再补测试”，而是严格的 TDD：

1. Red：先写或更新一个失败的测试，准确描述新需求、回归场景或接口契约变化
2. Green：只做让测试通过所需的最小实现
3. Refactor：在测试持续通过的前提下整理结构、命名和边界

## 硬性规则

- 任何接口改动都必须先写测试，再写实现。
- 这里的“接口改动”包括：
  - API 请求体或响应体结构变化
  - 鉴权、权限、状态码或错误语义变化
  - 流式响应行为变化
  - 对外可观察的数据副作用变化
  - 跨模块共享契约变化
- 路由相关变更优先把测试写在 `tests/api/`，除非已有更贴近的专用测试文件。
- 大改动、重构、架构调整，只有在以下验证全部通过后才能算完成：
  - `npm test`
  - 受影响主流程的 e2e 验证
- 如果受影响流程还没有现成 e2e，用例必须在同一任务中补齐或更新，不能把“没有 e2e”当作跳过理由。

## 完成标准

agent 在汇报完成时，必须明确说明：

- 先写了哪些测试
- 实现对应满足了哪些失败场景
- 跑了哪些 npm 测试
- 跑了哪些 e2e 验证

如果该改动本应经过 e2e 验证但未执行，这应视为阻塞项，而不是一句轻描淡写的备注。
